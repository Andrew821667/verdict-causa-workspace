# Ограниченные договорные контрфакты

## Назначение

Контрфактический слой отвечает на узкий вопрос: какое минимальное гипотетическое изменение проверенной конфигурации способно изменить формальный результат? Он не устанавливает факты, не предсказывает решение суда и не заменяет юридическую квалификацию.

Текущий срез относится к российскому институциональному пакету договорных отношений и работает только поверх типизированного `ObligationFactSet`.

## Архитектурная граница

Универсальное ядро определяет виды legal operators, модели изменений входа и результата, а также вычислительный бюджет. Пакет `contracts-ru-v0` определяет конкретные договорные операторы, русские юридические вопросы, preconditions и требования к доказательствам.

Исходный объект проверенных фактов неизменяем для анализа. Каждый сценарий получает отдельный hash гипотетической конфигурации и обязательную маркировку о том, что это не установленный факт.

## Библиотека v1

Версионированная библиотека содержит семь операторов:

1. изменить временную привязку исполнения;
2. активировать применимое основание освобождения;
3. квалифицировать завершенное исполнение как ненадлежащее;
4. добавить подтвержденную конфигурацию просрочки платежа;
5. проверить требование убытков с установленной причинной связью;
6. проверить требование убытков без установленной причинной связи;
7. проверить влияние указанного истечения срока исковой давности.

Операторы работают с формальными признаками. Например, оператор исковой давности не вычисляет срок самостоятельно и не подтверждает, что заявление стороны сделано надлежащим образом. Для этого нужны отдельные проверенные данные и экспертная оценка.

## Защитные ограничения

- Разрешено изменять только входные поля `ObligationFactSet`.
- Прямая запись в `breach_issue`, `damages_remedy_available` и другие результаты запрещена.
- Неизвестные или выдуманные факты отклоняются схемой.
- Каждый оператор требует human review и перечисляет необходимые виды доказательств.
- Preconditions проверяются до вычисления.
- Policy snapshot задает разрешение запуска, максимальное число сценариев и максимум изменений фактов в одной ветви.
- Дубли операторов и подмена hash библиотеки отклоняются.

## Материальность и ранжирование

После применения допустимой гипотезы формальная модель запускается заново. Сценарий считается материальным только при изменении хотя бы одного выходного поля. Критические сценарии ранжируются сначала по минимальному числу измененных фактов, затем по стабильному ID оператора.

Такое ранжирование показывает чувствительность текущей узкой модели, но не доказывает юридическую значимость гипотезы вне нее.

## Оценка

Отдельный benchmark содержит семь задач, по одной на каждый оператор. Red Team проверяет прямую подмену результата, неизвестный факт, отключение human review, обход бюджета фактов, обход бюджета сценариев, игнорирование preconditions и повторный запуск одного оператора.

Синтетический отчет формируется командой:

```bash
python scripts/export_synthetic_counterfactual_evaluation.py
```

Результат: `examples/synthetic_counterfactual_evaluation_report.json`.

## Ограничения Этапа 0

Это детерминированный одношаговый sensitivity-анализ. Здесь нет MCTS, LLM-генерации операторов, произвольного изменения графа, автоматического сбора доказательств и активации выведенных паттернов. Любое дальнейшее развитие остается bounded и проходит governance.
